## **A. PROJECT OVERVIEW (Questions 1-10)**

### **1. What is the main objective of your project?**

To develop an AI-powered system that automatically detects violent activities in surveillance videos using CNN-LSTM deep learning architecture and generates human-readable incident explanations using Generative AI.

---

### **2. Why is violence detection in surveillance videos important?**

Manual monitoring of surveillance systems is inefficient, prone to human error, suffers from operator fatigue, and doesn't scale with increasing camera counts. Automated violence detection enables faster incident response, consistent detection, and reduces the burden on security personnel.

---

### **3. What are the main components of your system?**

1. **React.js Frontend** - User interface for video upload and result visualization
2. **FastAPI REST API Backend** - Application logic and request orchestration
3. **CNN-LSTM ML Model** - Deep learning for violence detection
4. **GenAI Service** - Incident explanation generation
5. **PostgreSQL Database** - Data persistence and audit logging

---

### **4. What is the difference between your project and a simple CNN classifier?**

A simple CNN only analyzes individual frames without temporal context. Our CNN-LSTM architecture:
- **CNN**: Extracts spatial features from frames (what is in the scene)
- **LSTM**: Models temporal sequences (how the scene changes over time)
- This combination captures **motion patterns** essential for violence detection

---

### **5. Why did you use CNN-LSTM instead of 3D CNN?**

**Advantages of CNN-LSTM:**
- Lower computational requirements (suitable for CPU-only environments)
- Separates spatial and temporal feature extraction
- Easier to implement and understand for academic purposes
- Provides good balance between accuracy and computational efficiency

**3D CNNs** are more computationally expensive and typically require GPU acceleration.

---

### **6. What is Generative AI and why did you integrate it?**

Generative AI refers to AI models that can generate new content (text, images, etc.) based on learned patterns. We integrated GenAI to:
- Convert numerical detection results into human-readable incident reports
- Provide context and recommendations for security personnel
- Improve system interpretability and operational utility
- Generate confidence-aware explanations based on detection certainty

---

### **7. Does the GenAI process raw video data?**

**No.** The GenAI module operates as a **post-processing layer**. It receives:
- Violence probability scores (0.0-1.0)
- Classification result (VIOLENT/NON_VIOLENT)
- Confidence levels (HIGH/MEDIUM/LOW)
- Timestamps of key frames

It does **not** analyze video pixels directly. The CNN-LSTM handles video analysis; GenAI generates natural language explanations from the results.

## **B. DEEP LEARNING & ML (Questions 11-25)**

### **11. Explain how CNN works in your project.**

**CNN (Convolutional Neural Network)** extracts spatial features from individual video frames:

1. **Convolutional Layers**: Apply learnable filters to detect patterns (edges, textures, objects)
2. **Pooling Layers**: Reduce spatial dimensions while retaining important features
3. **Feature Maps**: Hierarchical representation of visual information
4. **Output**: High-dimensional feature vector representing frame content

We use a **custom lightweight CNN built from scratch** (two `TimeDistributed Conv2D` layers: 16 then 32 filters, 3×3 ReLU, each followed by MaxPool2D, then GlobalAveragePooling2D). No pre-trained ImageNet backbone is used — training from scratch keeps the model small, CPU-friendly, and fully controllable.

---

### **12. Explain how LSTM works in your project.**

**LSTM (Long Short-Term Memory)** models temporal sequences:

1. Receives sequence of CNN feature vectors (one per frame)
2. **Memory Cell**: Maintains information across time steps
3. **Gates**: Control information flow (forget, input, output gates)
4. **Hidden State**: Captures temporal patterns and motion dynamics
5. **Output**: Classification based on temporal sequence understanding

LSTM captures **how the scene changes over time**, essential for violence detection.

---

### **13. Did you use transfer learning? Why or why not?**

**Transfer Learning**: Using a pre-trained model as a starting point rather than training from scratch.

**No — we did not use transfer learning in this project.**

**Why we chose training from scratch:**
- The dataset is small (labeled short video clips) and domain-specific — ImageNet features may not transfer well
- Training from scratch keeps the model lightweight (Conv2D 16→32 filters, LSTM 64 units) and CPU-trainable in minutes
- Reduces external dependencies (no large pre-trained weight files to download)
- Academic transparency — every parameter in the model was learned on our own data

**Trade-off**: A larger dataset with a ResNet or MobileNet backbone would likely yield higher accuracy; the current design prioritises resource efficiency and academic clarity over benchmark performance.

---

### **14. What is the difference between training and inference?**

**Training:**
- Learning process where model adjusts weights based on labeled data
- Requires large datasets, GPU acceleration, significant time
- Not performed at runtime — training is done offline via `train.py` using the clips in `src/backend/data/`

**Inference:**
- Using a pre-trained model to make predictions on new data
- Does not update model weights
- Faster, can run on CPU
- What our system performs

**Our project**: Inference-only to ensure CPU-only feasibility for academic environments.

---

### **15. Why inference-only in production? How is the model trained?**

**The project includes an offline training script** (`src/backend/app/ml/train.py`) that builds and trains the CNN-LSTM model from scratch on labelled video clips. Training runs on CPU with `tensorflow-cpu`. At runtime, only inference is performed for efficiency.

**Reasons for separating training from the runtime path:**

1. **Deployment Simplicity**: Production deployment needs only the trained `.keras` model file, reducing runtime dependencies
2. **Resource Efficiency**: Inference is lightweight and runs on CPU; training is a one-time offline step
3. **Architectural Clarity**: Clear separation between the training pipeline (offline) and the inference pipeline (online)
4. **Reproducibility**: Training can be re-run with different data or hyperparameters independently of the application

**Trade-off**: A larger dataset with a ResNet or MobileNet backbone would likely yield higher accuracy; the current design prioritises resource efficiency, full architectural control, and academic transparency over benchmark performance.

---

### **16. How do you preprocess video data?**

**Video Preprocessing Pipeline:**

1. **Frame Extraction**: Extract frames at uniform temporal intervals (e.g., 1 frame per second)
2. **Frame Resizing**: Resize to CNN input dimensions (64×64, configured via `model_input_size`)
3. **Normalization**: Scale pixel values to [0, 1] range
4. **Channel Ordering**: Ensure RGB channel format
5. **Sequence Creation**: Create fixed-length sequences for LSTM input
6. **Batch Preparation**: Format for model inference

**Tools**: OpenCV for frame extraction, NumPy for array manipulation.

---

### **17. What is your model's input shape?**

**CNN Input**: (64, 64, 3) - Height × Width × Channels (RGB)

**LSTM Input**: (sequence_length, feature_dim)
- `sequence_length`: Number of frames (e.g., 30 frames)
- `feature_dim`: CNN feature vector size — **32** (output of `TimeDistributed GlobalAveragePooling2D` on `Conv2D(32)`)

**Example**: For 30-second video at 1 fps → 30 frames → LSTM input shape (30, 32)

---

### **18. How do you handle videos of different lengths?**

**Strategies:**

1. **Fixed Sampling**: Extract fixed number of frames uniformly distributed
2. **Padding**: Pad shorter sequences with zeros to match fixed length
3. **Truncation**: Truncate longer videos to maximum length
4. **Temporal Pooling**: Average features across excess frames

**Our approach**: Extract fixed number of frames (e.g., 30 frames) regardless of video duration, ensuring consistent input dimensions.

---

### **19. What is a confusion matrix and why is it important?**

**Confusion Matrix**: Table showing model predictions vs actual labels:

```
                Predicted Violent    Predicted Non-Violent
Actual Violent         TP                    FN
Actual Non-Violent     FP                    TN
```

**Importance:**
- **True Positives (TP)**: Correctly detected violence
- **False Negatives (FN)**: Missed violence (dangerous!)
- **False Positives (FP)**: False alarms
- **True Negatives (TN)**: Correctly identified non-violence

**For violence detection**: Minimizing False Negatives is critical (missing real violence).

---

### **20. What metrics evaluate your model?**

**Key Metrics:**

1. **Accuracy**: Overall correctness = (TP + TN) / Total
2. **Precision**: Of predicted violence, how many were correct = TP / (TP + FP)
3. **Recall**: Of actual violence, how many detected = TP / (TP + FN)
4. **F1-Score**: Harmonic mean of precision and recall
5. **Confidence Score**: Model's certainty in prediction (0.0-1.0)

**Priority**: **High Recall** (detect all actual violence) even if some false positives occur.

---

### **21. What is overfitting and how do you prevent it?**

**Overfitting**: Model memorizes training data but fails on new data.

**Prevention Techniques (if training):**
- **Dropout**: Randomly deactivate neurons during training
- **Regularization**: Penalize large weights (L1, L2)
- **Data Augmentation**: Artificially increase training data variety
- **Early Stopping**: Stop training when validation performance degrades

**Our project**: Uses a custom CNN trained from scratch; overfitting is controlled via dataset balance and early stopping during training.

---

### **22. What is the output of your CNN-LSTM model?**

**Raw Output**: Violence probability score between 0.0 and 1.0

**Example**: 0.87 means 87% confidence of violence

**Post-Processing:**
- **Threshold**: Apply threshold (e.g., 0.5) for binary classification
- **Confidence Mapping**:
  - Score ≥ 0.8 or Score ≤ 0.2 → HIGH confidence (strong positive or strong negative detection)
  - Score ≥ 0.6 or Score ≤ 0.4 → MEDIUM confidence (moderate evidence)
  - 0.4 < Score < 0.6 → LOW confidence (uncertain/ambiguous)
- **Classification**: VIOLENT if score ≥ threshold, else NON_VIOLENT

---

### **23. How does your system handle false positives?**

**Strategies:**

1. **Confidence Levels**: Report confidence (HIGH/MEDIUM/LOW) to indicate certainty
2. **GenAI Explanations**: Explain detection and suggest manual review for medium/low confidence
3. **Threshold Tuning**: Adjust detection threshold based on operational requirements
4. **Human-in-the-Loop**: System recommends manual verification for uncertain detections

**Trade-off**: Balance between catching all violence (high recall) vs. minimizing false alarms.

## **D. SYSTEM ARCHITECTURE & IMPLEMENTATION (Questions 36-50)**

### **36. What is JWT and why did you use it?**

**JWT (JSON Web Token)**: Compact, URL-safe token format for secure information transmission.

**Structure**: `header.payload.signature`
- **Header**: Token type and signing algorithm
- **Payload**: User claims (user_id, role, expiration)
- **Signature**: Cryptographic signature for verification

**Why JWT:**
- **Stateless**: Server doesn't store session data
- **Scalable**: No server-side session storage required
- **Secure**: Cryptographically signed and optionally encrypted
- **Standard**: Industry-standard authentication mechanism
- **API-Friendly**: Easily transmitted in HTTP headers

---

**Two Roles:**

**USER Role:**
- Upload videos
- View own analysis results

- View own history

**ADMIN Role:**

**Implementation**: Role stored in database, verified on every API request using JWT token claims.

---

### **38. How does authentication work in your system?**

**Authentication Flow:**

1. **Registration**: User provides username, email, and password
   - Password hashed with bcrypt (12 rounds, `$2b$`)
   - User created with default USER role

2. **Login**: User submits credentials
   - Verify username exists
   - Verify password hash matches
   - Generate JWT access token (expiry: 30 minutes)
   - Return token to frontend

3. **Authenticated Requests**: Frontend includes JWT in Authorization header
   - Backend verifies token signature
   - Extracts user_id and role from token
   - Grants/denies access based on role

### **41. Why did you use psycopg2 for database access instead of an ORM?**

**psycopg2** is a low-level PostgreSQL adapter for Python that executes raw SQL directly, without an ORM abstraction layer.

**Reasons for choosing direct SQL:**
- Full control over SQL queries and query plans
- No ORM overhead or abstraction complexity
- Transparent, auditable SQL for academic review
- Parameterized queries provide built-in SQL injection prevention
- Lightweight: no additional ORM dependency required

**ORM (Object-Relational Mapping)** such as SQLAlchemy maps tables to Python classes, which adds convenience but also complexity. For this project, direct SQL via psycopg2 is simpler, more transparent, and sufficient for the schema size.

**Connection Management**: psycopg2 connections are managed per-request and closed after use to prevent resource leaks.

### **43. What is CORS and why is it important?**

**CORS (Cross-Origin Resource Sharing)**: Browser security feature that restricts web pages from making requests to different domains.

**Problem**: React frontend (localhost:3000/5173) calling FastAPI backend (e.g. localhost:8000) = different origins

**Solution**: Configure FastAPI backend to allow requests from frontend origin (CORSMiddleware)

**Implementation**:
```python
from fastapi.middleware.cors import CORSMiddleware
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.cors_allowed_origins,
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "DELETE", "OPTIONS"],
    allow_headers=["Authorization", "Content-Type", "X-Request-ID"],
)
```

**Production**: Configure specific allowed origins (not wildcard `*`).

### **64. Explain transfer learning in depth.**

**Transfer Learning**: Leverage knowledge from one task (source) to improve performance on another task (target).

**Process:**

1. **Pre-training**: Model trained on large dataset (e.g., ImageNet - 1M images)
2. **Feature Extraction**: Early layers learn general features (edges, textures, shapes)
3. **Transfer**: Use pre-trained model as feature extractor for new task
4. **Fine-tuning** (optional): Adjust some layers on new dataset

**Why It Works:**
- Low-level features (edges, colors) are universal
- Reduces need for large labeled datasets
- Faster training
- Better performance with limited data

**Our Implementation**: Custom CNN built from scratch (two `TimeDistributed Conv2D` layers, 16→32 filters, GlobalAveragePooling2D → feature dim 32), followed by LSTM 64 units. No transfer learning or pretrained weights.

### **67. Explain gradient descent and backpropagation.**

**Gradient Descent**: Optimization algorithm to minimize loss function by iteratively adjusting model weights.

**Process:**
1. Initialize random weights
2. Forward pass: Make predictions
3. Calculate loss (difference between predictions and true labels)
4. Compute gradients (rate of change of loss w.r.t. weights)
5. Update weights: `weight = weight - learning_rate * gradient`
6. Repeat until convergence

**Backpropagation**: Algorithm to compute gradients efficiently by propagating errors backward through network layers.

**Relevance**: Used during model training (not in our inference-only project, but understanding is important).

---

### **68. What is batch normalization and why is it useful?**

**Batch Normalization**: Normalizes layer inputs across a mini-batch during training.

**Process:**
1. Calculate mean and variance of mini-batch
2. Normalize inputs to zero mean, unit variance
3. Apply learnable scale and shift parameters

**Benefits:**
- Faster training convergence
- Higher learning rates possible
- Reduces internal covariate shift
- Mild regularization effect

**Our Project**: The CNN-LSTM is trained from scratch; the architecture does not include batch normalization layers. The model uses Conv2D with ReLU activation followed by MaxPooling and GlobalAveragePooling2D.

### **69. How would you handle class imbalance in violence detection?**

**Problem**: Much more non-violent data than violent data in real-world scenarios.

**Solutions:**

1. **Resampling**:
   - **Oversampling**: Duplicate violent examples
   - **Undersampling**: Reduce non-violent examples
   - **SMOTE**: Synthetic Minority Oversampling Technique

2. **Class Weights**: Assign higher loss weight to minority class (violence)

3. **Focal Loss**: Focus learning on hard, misclassified examples

4. **Data Augmentation**: Generate synthetic violent examples through transformations

5. **Threshold Adjustment**: Lower classification threshold for violence detection

6. **Evaluation Metrics**: Use precision-recall instead of accuracy

**Our Project**: The CNN-LSTM is trained from scratch using class weights to handle imbalance. In `train.py`, class weights are computed dynamically based on label distribution, assigning higher loss weight to the minority class.
